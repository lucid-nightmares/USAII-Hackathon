# Role 4: Demo Director

Using the previous three review files as constraints.

## 90-second demo storyline
1. Start on the hero and say the value proposition in one sentence: SignalSafe scans locally and only sends a redacted incident packet when a parent needs to act.
2. Run a scan for Maya and open the highest-risk incident.
3. Show the AI reasoning panel and the raw-on-device versus parent-visible packet comparison.
4. Acknowledge the incident and show the lifecycle strip advance.
5. Trigger guarded reveal to prove that deeper access is exceptional and audited.
6. Take a parent action like pause or backup review and show route state plus audit log change.
7. Switch to Controls to show dignity mode, approval queue, and integrity state.
8. End on the trust boundary and action history as proof that this is not surveillance software.

## 3-click version of the demo
1. Run device scan.
2. Open selected incident and trigger guarded reveal or acknowledgement.
3. Take a parent action and then open Controls to show the audit trail and policy response.

## 1 wow moment
Guarded reveal with audit trail: the parent starts with a redacted packet, requests more context only when justified, and the system logs that privacy boundary crossing in real time.

## 1 trust moment
Child dignity mode visibly downgrades uncertain incidents into check-in-first guidance rather than lock-first behavior.

## 5 things that currently weaken the demo
1. Too many screens still read as seeded dashboards rather than causally connected system views.
2. There is not yet a reveal mechanic that proves the privacy model live.
3. Action history exists, but it does not yet clearly show data-access transparency.
4. Integrity state is visible but does not yet affect the main incident story enough.
5. Confidence is shown, but not yet obviously used to change guardian options.

## 5 concrete ways to make statefulness visible
1. Persist incident lifecycle state after each guardian action.
2. Add a guarded reveal button that updates the incident, audit log, and trust boundary card.
3. Change route state after acknowledgement and escalation actions.
4. Record policy-driven restrictions and allow those restrictions to affect current incident actions.
5. Add a degraded coverage or tamper event that changes both controls and incident handling.
